# kimex — Kasten Import Key Extractor

`kimex` retrieves the **import keys** (`receiveString`) from
[Kasten K10](https://docs.kasten.io/) policies and exports them in a
structured way.

## Why?

During a K10 migration/restore, the **import policies** on the receiving
cluster need the import key (`receiveString`) generated by the **export
policies** on the source cluster. That key is buried in the policy CRD and is
not trivial to extract.

`kimex` addresses two needs:

1. **Make it easy to create import policies** on a receiving cluster by
   clearly exposing `receiveString`, `profile` and `migrationToken`.
2. **Escrow this sensitive data** so you can restore even without the
   original K10 catalog.

## How it works

- Target CRD: `policies.config.kio.kasten.io` (`v1alpha1`)
- Import key: `spec.actions[].exportParameters.receiveString`
- Only present on export actions with `exportData.enabled=true`
- Associated references kept:
  - `exportParameters.profile` (`name` + `namespace`)
  - `exportParameters.migrationToken` (`name` + `namespace`)

Parsing is done via `... -o json | jq` (no fragile `grep`/`sed` on YAML).

### Environment detection

- If **OpenShift** is detected (presence of the `route.openshift.io` or
  `config.openshift.io` API groups), `kimex` uses **`oc` exclusively**.
- Otherwise, it falls back to `kubectl`.
- Detection runs **per context** (each cluster may differ).

## Requirements

- `jq`
- `oc` **or** `kubectl` (ideally both in a mixed environment)

Dependencies are checked at startup; the script fails cleanly if any are
missing.

## Installation

```bash
chmod +x kimex.sh
# optional: put it on your PATH
sudo ln -s "$(pwd)/kimex.sh" /usr/local/bin/kimex
```

## Options

| Option | Description |
|--------|-------------|
| `-n, --namespace <ns>` | K10 namespace (default: `kasten-io`). |
| `-c, --context <ctx>` | Kubeconfig context. Repeatable or comma-separated. |
| `--context-file <f>` | File with one context per line (`#` = comment). |
| `--all-contexts` | Iterate over all contexts in the kubeconfig. |
| `--kubeconfig <path>` | Specific kubeconfig. |
| `-o, --output <fmt>` | `table` (default), `json` or `csv`. |
| `-d, --output-dir <dir>` | Write one `.key` file per policy (receiveString only). |
| `--dry-run` | Show the plan (contexts, commands, `.key` files) without contacting any cluster or writing any file. |
| `-h, --help` | Help. |

## Examples — single cluster

```bash
# Current context, default namespace, table output
./kimex.sh

# Custom namespace, JSON output
./kimex.sh -n kasten-io -o json

# A specific context + escrow keys to disk
./kimex.sh -c prod-cluster -d ./keys

# Dedicated kubeconfig
./kimex.sh --kubeconfig /etc/k10/admin.conf -c staging -o json
```

## Examples — multi-cluster

```bash
# Multiple contexts (comma-separated)
./kimex.sh -c prod-a,prod-b -o json

# Multiple contexts (repeated flag)
./kimex.sh -c prod-a -c prod-b -d ./keys

# Context list from a file
cat > clusters.txt <<EOF
# production clusters
prod-a
prod-b
EOF
./kimex.sh --context-file clusters.txt -d ./keys

# All contexts in the current kubeconfig
./kimex.sh --all-contexts -o json
```

Each result is **tagged with its source context/cluster**. An unreachable
cluster, a missing namespace or an invalid context is logged and then
**skipped**: the other clusters keep being processed.

## Output formats

- **table** (default): readable overview. The `receiveString` is **truncated**
  here (full value via `--output json`, `--output csv` or `--output-dir`).
- **json**: full array of objects, ideal for automation.
- **csv**: one row per policy with the **full** `receiveString` (header
  included, escaping handled). Ideal for spreadsheets / scripts:
  `./kimex.sh -c prod-a -o csv > import-keys.csv`
- **`.key` files** (`--output-dir`): one file per policy named
  `<context>__<namespace>__<policy>.key`, containing **only** the
  `receiveString` (direct copy/paste). Created with `chmod 600`.

Example JSON object:

```json
{
  "context": "prod-a",
  "policy": "mysql-export",
  "namespace": "kasten-io",
  "receiveString": "<migration key>",
  "profile": "azure-blob",
  "profileNs": "kasten-io",
  "migrationToken": "mysql-export-token",
  "migrationTokenNs": "kasten-io"
}
```

## ⚠️ Security

The `receiveString` is **sensitive data** (a migration key). Anyone who holds
it can import your K10 data.

- **Never commit** `kimex` output.
- The bundled `.gitignore` covers `*.key` and the usual `--output-dir`
  directories (`keys/`, `output/`, etc.).
- `.key` files are created with `chmod 600`.
- Store escrowed keys in a vault (Vault, a secret manager…), not in plain
  text in a repository.

Check before pushing:

```bash
git status --ignored
```
